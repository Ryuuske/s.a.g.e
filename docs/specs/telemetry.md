# Loop Telemetry

The per-turn governance-metrics surface for the S.A.G.E. orchestrator loop.

The orchestrator emits one JSONL row per agent turn — every specialist dispatch, every audit verdict, every self-check pass — to `~/.sage/telemetry/turns.jsonl`. The miner later ingests the file into the nook as wing `telemetry`, hall `turns`, so existing recall paths can query it without bespoke tooling.

## Why this exists

Without per-turn telemetry, the framework gives no empirical answer to:

- How often do the dual-auditor pairs disagree?
- How often does the User reject a plan after specialist consultation?
- How often does the same blocking finding recur across turns?
- Does mode-routing actually fire when it should?
- Are aidev sessions trending shorter or longer than the prior week?

The integration spec calls this out as "invisible — no signal at all" without telemetry.

## Row schema

Each line of `turns.jsonl` is one JSON object with these fields (defined in `src/sage_mcp/telemetry.py::TurnRecord`):

| Field | Type | Notes |
| --- | --- | --- |
| `turn_id` | string (12 hex chars) | Same `turn_id` for paired-auditor rows. Generated by `new_turn_id()`; orchestrator can pin it across calls. |
| `timestamp` | ISO 8601 UTC | `YYYY-MM-DDTHH:MM:SS.mmmZ` (millisecond precision; the extra precision avoids same-second collisions on paired-auditor rows). |
| `phase` | enum | `plan` \| `dispatch` \| `audit` \| `implement` \| `commit` \| `self-check`. |
| `mode` | enum | `aidev` \| `normal`. |
| `agent` | string | Agent that produced the verdict, or `orchestrator`. |
| `verdict` | enum \| null | `APPROVE` \| `REQUEST_CHANGES` \| `REJECT` \| `HOLD` \| `ABORT` \| `null` (for `phase != audit`). |
| `severity_top` | integer | Max severity across this turn's findings. `0` if none. |
| `findings_count` | integer | Number of findings in the verdict block. |
| `adr_produced` | string \| null | Path to an ADR produced this turn, or `null`. |
| `wing` | string \| null | The current wing at turn time, if `current_wing` was set. |
| `extras` | object | Forward-compatibility slot; not parsed by current consumers. |

## How rows are written

The orchestrator calls one of:

- `telemetry.log_turn(TurnRecord(...))` — full control.
- `telemetry.log_from_verdict(parsed_verdict, phase=..., mode=...)` — convenience for the common audit-verdict case. Returns the `turn_id` written.

Both swallow I/O failures. Telemetry must not break the orchestrator; the cost of a missing row is a small gap, the cost of a crashed turn is the User's session.

## How rows are mined into the nook

The miner is taught to walk `~/.sage/telemetry/turns.jsonl` and file each row as a drawer with:

- `wing = "telemetry"` (must be registered in `wing_config.json`)
- `hall = "turns"`
- `agents = ["orchestrator"]`
- drawer content = the raw JSON line, verbatim
- `metadata.filed_at` = the row's `timestamp`
- `metadata.turn_id`, `metadata.phase`, `metadata.verdict` for filterable recall

After mining, queries like `sage recall --wing telemetry --agent aidev-code-reviewer` surface the audit rows.

The mining step is opt-in via `sage mine ~/.sage/telemetry/`. Auto-mining on session end is intentionally not wired in — telemetry rows are cheap to keep on disk and the operator may want to truncate or rotate the file periodically.

## Privacy

Rows written by **`sage verdict log`** (`log_from_verdict`, the path the autonomy loop and §16 use) contain only the bounded schema fields above — `agent` (lane name), `verdict`, integer `severity_top`, `findings_count`, `wing`, `adr_produced`, `phase`, `mode`, `turn_id`, `timestamp`. They do NOT store the per-finding `file` paths or `summary` text the verdict block carries (`log_from_verdict` records the aggregate `findings_count` and the top severity, not the individual `@@FINDING` file/line/summary fields), and they do NOT contain User content, code excerpts, or transcript data. The verdict parser rejects free-form fields outside the documented set.

The lower-level `log_turn(TurnRecord(...))` writer carries an `extras` dict (the forward-compatibility slot) that `sage verdict log` never populates. A direct `log_turn` caller is responsible for keeping `extras` free of User content — that boundary is the caller's, not the parser's; `sage verdict log` rows are the bounded set described above.

Per CLAUDE.md design principle "privacy by architecture": the telemetry file never leaves the machine. It is not pushed to GitHub (covered by the standard ignore patterns; `~/.sage/` lives outside any repo).

## Inspection

Quick tail:

```bash
tail -n 20 ~/.sage/telemetry/turns.jsonl | jq .
```

Or via Python:

```python
from sage_mcp.telemetry import read_recent
for row in read_recent(limit=20):
    print(row["timestamp"], row["agent"], row["verdict"])
```

## Health check

A future `sage audit telemetry-health` subcommand surfaces:

- Gaps in coverage (sessions where no rows were written despite specialist dispatch).
- Parser errors over the window.
- Mode-routing anomalies (rows tagged `aidev` mode where the agent name is non-aidev, or vice versa).

For now those checks can be ad-hoc via `read_recent()` in a one-off script.
